Deuda técnica

La deuda técnica se la falta de calidad en el código que escribimos y repercute en el coste (tiempo, dinero) futuro

Esquema de deuda técnica de Martin Fowler

  • Imprudente:

    "No hay tiempo, solo copia y pega eso"

  • Inadvertida:

    "¿Qué son los patrones de diseño?"

  • Prudente:

    "Tenemos que entregar rápido, ya refactorizaremos"

  • Prudente e Inadvertida:

    "Ahora sabemos como lo deberiamos haber hecho"

¿Cómo se paga la deuda técnica?

Refactorización

Es un proceso que tiene como objetivo mejorar el código sin alterar su comportamiento para que sea más entendible y tolerante a cambios, las pruebas unitarias o automaticas son imprescindibles en estos procesos.

Nombres pronunciables y expresivos

  • Variables en inglés
  • Las variables deben indicar que cosa van a guardar

    • Ejemplo: const numberOfUnit: 54;

Ausencía de información técnica en los nombres

Las clases o interfaces no deben tener expresiones técnicas

Ejemplo: AbstractUser (mal) - User (bien)

Nombres según el tipo de datos

Arrays

const fruit = ['manzana', 'pera'] (mal)

const fruitList = ['manzana', 'pera'] (regular)

const fruits = ['manzana', 'pera'] (bueno)

const fruitNames = ['manzana', 'pera'] (mejor)

Booleans

const fruits = ['manzana', 'pera'] (bueno)

const open = true (mal)

const isOpen = true (bien)

Utilizar, is, can, etc... utilizar verbos para ser más claros

Numbers

const cars = 10 (mal)

const totalOfCars = 5 (bien)

Funciones

createUserIfNotExist(); (mal)

createUser(); (bien)

La función o el método debe determinar si el usuario existe o no, y según el caso, crearlo o no, por lo tanto el "IfNotExist" no nos aporta nada.

Clases

  • El nombre es lo más importante de una clase.
  • Formados por un sustantivo o frases de sustantivos.
  • No deben de ser muy genéricos.
  • Usar UpperCamelCase

3 preguntas para determinar saber si es un buen nombre

  • ¿Qué exactamente hace la clase?
  • ¿Cómo exactamente esta clase realiza cierta tarea?
  • ¿Hay algo específico sobre su ubicación

class Manager{}; (mal)

Funciones

Se recomienda no tener más de 3 parametros, pasar un objeto

Otras recomendaciones

  • Simplicidad es fundamental
  • Funciones de tamaño reducido
  • Funciones de una sola línea sin causar complejidad
  • Funciones de menos de 20 líneas
  • Evita el uso del "else"
  • Prioriza el uso de la condicional ternaria

Principio DRY

  • Simplemente es evitar tener duplicidad de código.
  • Simplifica las pruebas.
  • Ayuda a centralizar los procesos.
  • Aplicar el principio DRY, usualmente lleva a refactorizar.
